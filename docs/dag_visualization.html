Understanding Directed Acyclic Graphs (DAGs)

What is a DAG?

A Directed Acyclic Graph is a conceptual representation of a series of activities. The order of the activities is depicted by a graph, which is visually presented as a set of circles (nodes) and lines (edges).

  • Directed: Each edge has a defined direction (an arrow), meaning the relationships between nodes are one-way (from A to B).
  • Acyclic: It is impossible to start at one node and follow a consistently-directed sequence of edges that loops back to the same node. There are no cycles.
  • Graph: The overall structure made up of interconnected vertices (nodes) and edges.

DAGs are heavily used in data processing pipelines, task scheduling, compilation dependencies, and version control systems like Git.

Topological Sorting

Topological sorting is a linear ordering of a DAG's vertices such that for every directed edge U → V, vertex U comes before V in the ordering.

In the viewer above, each node displays its current in-degree. When a node's in-degree reaches 0, it turns yellow and appears in the queue panel. The algorithm repeatedly takes the next queued node, appends it to the output order, and subtracts 1 from each downstream node's in-degree.

In the interactive visualization above, we use Kahn's Algorithm for the topological sort:

  1. Compute the in-degree (number of incoming edges) for each node.
  2. Identify all nodes with an in-degree of 0 (no dependencies) and add them to a queue.
  3. While the queue is not empty:
    • Remove a node from the queue (mark it as "done" or "processed").
    • For each outgoing edge from this node, reduce the in-degree of the destination node by 1.
    • If the destination node's in-degree becomes 0, add it to the queue.

This ensures tasks are only executed when all their prerequisite dependencies have been met.

The Thermal-Perigee Pipeline Architecture

In this project (internal/dag/engine.go), the data extraction pipeline is structured as four primary processing stages coordinated by an engine scheduler:

  • Collectors: The ingestion points. They scrape APIs, OSINT sources, or passive DNS databases to gather initial raw data (like domains or IPs).
  • Enrichers: They augment the collected assets. For example, resolving a domain to an IP, looking up BGP ASN data, or finding reverse DNS records.
  • Scheduler: It is not a processing stage. It decides whether newly discovered domains become the next collection frontier, keeping control flow in the engine rather than in the nodes themselves.
  • Filters: Data sanitation sequence. Dead domains, out-of-scope assets, or honeypots are dropped.
  • Exporters: Formatting and sink stage. Cleaned data is dispatched to CSV, JSON, or a database.

Each processing wave stays acyclic. If enrichment discovers new domains, the scheduler starts a later collection wave instead of introducing direct node-to-node cycles. By decoupling these stages using interfaces, the architecture remains highly modular and allows tasks to be executed efficiently in parallel.
